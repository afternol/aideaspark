## エクスポート

### html2canvasはTailwind v4のoklch/lab色を解析できない（2026-04-04）
- **状況**: カスタマイズ版のPDFエクスポートで `html2canvas` が `Attempting to parse an unsupported color function "lab"` エラー
- **原因**: Tailwind v4は内部で `oklch()` / `lab()` 色関数を使用。html2canvas 1.4.1はこれらに非対応。ページ内のDOM要素をキャプチャする際、親要素から継承されるTailwindのCSS変数が解析に失敗する
- **解決**: `iframe` を使って完全に隔離した環境を作成。iframe内にTailwind CSSが一切漏れない純粋なHTMLドキュメント（hex色のみ）を書き込み、その `body` をhtml2canvasでキャプチャ
- **教訓**: html2canvasを使う場合、親ページのCSSフレームワークの色形式に注意。最新CSSカラー関数（oklch, lab, lch等）は非対応。iframe隔離が安全策

### PDFのページ境界で文字が切れる問題（2026-04-04）
- **状況**: jsPDFで複数ページのPDFを生成すると、297mm間隔の機械的スライスでテキストの途中で切断される
- **解決**: キャンバスのピクセル行をスキャンし、改ページ候補位置の前後±40mm範囲で「最も白い行」（セクション間の空白）を探してそこで分割。各スライスを個別キャンバスに切り出してPDFに配置。2ページ目以降は上部に8mmマージン追加
- **教訓**: html2canvas + jsPDFの複数ページ出力では、単純な固定高さスライスは使えない。空白行検出アルゴリズムを入れることでテキスト切れを防止できる

### Word出力にはdocxライブラリが最適（2026-04-04）
- **状況**: クライアントサイドでWord文書を生成する必要があった
- **選定**: `docx` ライブラリ + `file-saver`。テキストとしてコピー・編集可能なネイティブ.docxを生成
- **注意点**: `file-saver` には `@types/file-saver` の型定義が別途必要（ないとNext.jsのビルドでType error）
- **教訓**: PDF（画像ベース）とWord（テキストベース）は別々の生成ロジックが必要。Wordはdocxライブラリで構造化ドキュメントを直接組み立てるのが最もクリーン

## UI/UX

### ツールチップの z-index 問題（2026-04-02）
- **状況**: トレンドヒートマップのタイルにホバーで表示されるツールチップが、隣接タイルに隠れて読めない
- **原因**: `flex-wrap` で複数行になるタイル群で、後の行のタイルが前の行より z-index が高くなるため、上に表示するツールチップが被る
- **解決**: タイル自体に `hover:z-50`、ツールチップに `z-[100]` を付与。Tailwind の `group/name` でネスト対応
- **教訓**: `position: relative` な要素が並ぶ場合、ツールチップは要素単体の z-index だけでなく、hover 時に親のスタッキングコンテキストを引き上げる必要がある

### ブックマーク（localStorage）vs コレクション（DB）の二重管理（2026-04-02）
- **状況**: カード上の保存ボタンが `useBookmarkStore`（localStorage）で、コレクション機能は API 経由で DB に保存。2つが別系統で同期しない
- **解決**: 全画面の保存ボタンを `AddToCollection` コンポーネントに統一。compact プロップでアイコンのみ表示に対応
- **教訓**: 「とりあえず localStorage で」の仮実装は、後から DB 版との統一コストが全コンポーネントに波及する。保存系機能は最初から永続化先を統一すべき

### エンゲージメントボタンはカード一覧にも出す（2026-04-02）
- **状況**: リアクション（いいね・気になる・参考になった）が詳細ページにしかなく、一覧から押せない
- **解決**: `IdeaSummaryCard` にリアクションボタンを埋め込み、API に種類別カウント + ユーザーリアクション状態を返す拡張を実施
- **教訓**: エンゲージメント導線は「見る場所 = 押せる場所」にする。一覧カードにアクションがないと詳細を開く動機が薄い

### 不要な情報の削除判断（2026-04-02）
- **状況**: アイデア詳細のタグ行に「難易度: 低/中/高」バッジがあったが、フィルター検索には使われておらず、6軸スコアで実質カバーされていた
- **解決**: 指摘を受けて即削除
- **教訓**: 表示項目は「ユーザーがその場で意思決定に使うか」で判断。飾りの情報はノイズになる

## フィルター・検索

### ランキングにもフィルターが必要（2026-04-02）
- **状況**: ランキングが全件一律表示で、自分の興味領域のランキングが見られない
- **解決**: 領域・対象業界・対象顧客の3軸 SearchableSelect フィルターを追加
- **教訓**: 一覧系ページには必ず「自分に関係あるものだけ見る」手段を用意する

### 比較ページの選択 UX（2026-04-02）
- **状況**: ドロップダウン式 SearchableSelect で比較対象を選ぶ UI が、アイデア数が多いと探しづらい
- **解決**: キーワード検索 + カード型ピッカーに変更。サービス名・カテゴリ・業界・タグで検索可能に
- **教訓**: 選択肢が多い場合、ドロップダウンよりカード型ピッカーの方が情報量（バッジ等）を提示でき探しやすい

## API 設計

### 一覧 API に集計情報を含める（2026-04-02）
- **状況**: `/api/ideas` が総リアクション数しか返さず、種類別カウントとユーザー自身のリアクション状態がなかった
- **解決**: `reactionsByType` の groupBy と、`sessionId` による `userReactions` を追加
- **教訓**: カードにインタラクティブ要素を置くなら、一覧 API がその状態を返す必要がある。N+1 を避けるためバッチ取得で設計する

## AI機能

### 汎用AIとの差別化を常に問う（2026-04-02）
- **状況**: AIアイデア壁打ち機能を検討した際、「ChatGPTと何が違うのか」と指摘を受けた
- **教訓**: BizIdea独自のデータ（トレンドスコア・6軸評価・構造化DB・閲覧履歴）を活用しない限り、汎用AIと同じ機能を作っても価値がない
- **原則**: AI機能は「考えて終わり」ではなく「プラットフォーム内のアクション（プラン生成・比較・コレクション保存）に繋がる」設計にする
- **実例**: AI検索は全アイデアDBをコンテキストに渡し、既存データに基づいたマッチングを行う。汎用AIにはできない精度

### AI検索のサジェストは万人向けにする（2026-04-02）
- **状況**: 初期サジェストが「AIエージェントを使った業務効率化ビジネス」等テック寄りだった
- **修正**: 「一人で始められるスモールビジネス」「リピート収益が見込めるストック型ビジネス」等、起業検討者なら誰でもピンとくる表現に
- **教訓**: サジェストはターゲット層の最大公約数的な関心事にする

### AIモデルはコスパで選ぶ（2026-04-02）
- Claude Haiku 4.5を採用（入力$0.80/1M、出力$4.00/1M）
- 構造化データからのマッチング＋JSON出力にはHaikuで十分
- 1回あたり約0.1〜0.2円、レスポンス1〜2秒

## 機能の取捨選択

### 再カスタマイズは選択中バージョンをベースにする（2026-04-04）
- **状況**: マイアイデアの「再カスタマイズ」ボタンが元アイデア詳細ページへのリンクだった。複数バージョンがあっても常にオリジナルからやり直しになる
- **解決**: マイアイデア画面内にインラインのカスタマイズフォームを追加。選択中のバージョン（`customIdeaId`）をベースにAPIリクエストを送信
- **教訓**: バージョン管理UIがある場合、アクション（再カスタマイズ・プラン生成等）は「今選択中のバージョン」に対して実行されるべき。別ページに飛ばすとコンテキストが失われる

### 「時期尚早」な機能は入れない（2026-04-02）
- **フォロー/フォロワー**: コミュニティが成熟する前にソーシャル機能を入れても使われない → 全削除
- **「やります」宣言**: エンゲージメントとして重すぎた → 削除し、軽いリアクション（いいね等）に集約
- **教訓**: 機能は「今のユーザー数・利用頻度で意味があるか」で判断。将来のための先行実装は負債になる

### プロフィール公開/非公開は最初から入れる（2026-04-02）
- プライバシー設定は後から追加すると既存ユーザーのデフォルトが問題になる
- 最初から`profilePublic: true`のデフォルトと切替UIを実装しておく

## レイアウト

### stickyが効かない時はoverflow-autoを疑う（2026-04-02）
- **状況**: 左サイドバーのstickyが効かない
- **原因**: 親の`<main>`に`overflow-auto`が設定されていた。stickyは最も近いスクロール可能な祖先に対して機能する
- **解決**: `overflow-auto`を削除。ページ全体のスクロールはbody側で行う

### 左右パネルの視覚的分離（2026-04-02）
- 背景色の差（左: bg-muted/30、右: 白）+ border-rで境界を明確にする
- 左サイドバーは画面端まで伸ばす（親レイアウトの余白を負マージンで打ち消し）
